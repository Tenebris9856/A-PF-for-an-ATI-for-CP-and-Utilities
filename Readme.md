# Методология оформления *текстового интерфейса* для *консольных программ* и *утилит*
## Структура *программы*:
- **`Заглавие`** — **наименование *программы***, *кратко описывающее **основную цель** или **задачу***:
    - **`Описание`** (**&laquo;`|}…`&raquo;**) — **полное описание *программы***:
        - **`Основное содержимое`** — **основная часть *программы***:
            - **`Программный вывод`**[^1] (**&laquo;`|>-…`&raquo;**) — **сообщение *программы* о *выполненных инструкциях***;
            - **`Системное сообщение`**[^1] (**&laquo;`|}-…`&raquo;**) — **сообщение *разработчика***, *адресованное **пользователю** от **лица программы***;
            - **`Авторское сообщение`** (**&laquo;`| -…`&raquo;**) — **сообщение *разработчика***, *адресованное **пользователю** от **собственного лица***;
            - **`Ввод от пользователя`** (**&laquo;`|}>-…`&raquo;**) — **ввод от *пользователя***.
- **`Завершение программы`** — **сообщение о *завершении программы***, *а также **авторство** или **право собственности***:
    - **`Сообщение о завершении программы`** (**&laquo;`/…`&raquo;**);
    - **`Авторство или правообладатель`** (**&laquo;`/>…`&raquo;**);
    - **`Сообщение о способе закрытия окна программы`** (**&laquo;`/>-…`&raquo;**).

> [!NOTE]
> **Примечание**: ***все функциональные программные модули** разделяются **разделительной линией*** (***последовательность знака равно*** — ***&laquo;`===`&raquo;***); *если внутри **модуля** предусмотрено **заглавие** и **содержимое***, ***они** разделяются **малой разделительной линией*** (***последовательность знака дефиса*** — ***&laquo;`---`&raquo;***).

> [!IMPORTANT]
> **Примечание**: ***длина текстовой строки** зависит от **длины `разделительной линии`***, ***которая** в **свою очередь** зависит от **длины `заглавия программы`***; ***все переносы слов** строго фиксируются по **длине `разделительной линии`***; *однако*, *если **длина текста** заведомо превышает **длину `разделительной линии`***, ***разделительная линия** должна быть продолжена на **четыре*** (***4***) ***символа последовательностью знака дефиса*** (***&laquo;`---`&raquo;***).

 [^1]: **Примечание**: *допускается **интеграция программного вывода** и **системного сообщения***.

## Архитектура *библиотеки текстового интерфейса*
### Общая *модель*
Программа построена на модели &laquo;конвейер с отложенной подстановкой&raquo;: весь интерфейс сначала полностью собирается в одну строку, затем обрабатывается фреймворком (парсинг, декомпозиция, адаптивный перенос, сборка), и только после этого выводится пользователю. Маркеры-заглушки ("`\ввод`" и пользовательские маркеры данных) позволяют прерывать вывод и подставлять значения после взаимодействия с пользователем.

### Уровни *архитектуры*
Программа разделена на четыре уровня, от верхнего к нижнему:
- **Уровень № 1** — точка входа (строки 26–75); управляет жизненным циклом программы: вызывает рендер, обрабатывает маркеры ввода, разбивает готовую строку на фрагменты &laquo;до ввода&raquo; и &laquo;после ввода&raquo;, передаёт управление пользовательской логике, выводит результат; здесь же размечены зоны для пользовательских переменных и обработки ответов;
- **Уровень № 2** — композиция (строки 77–92); декларативное описание структуры интерфейса: что за чем идёт; вызывает алгоритмы рендеринга блоков и склеивает их результаты в единую строку; пользователь работает только на этом уровне — описывает свою программу через вызовы рендеров;
- **Уровень № 3** — библиотека рендеринга (строки 94–221); набор алгоритмов-функций, каждый из которых превращает содержимое в размеченный блок согласно спецификации; все рендеры следуют единому протоколу: возвращают строку, завершённую граничным маркером новой строки с пробелом;
- **Уровень № 4** — фреймворк (строки 208–443); движок обработки: принимает сырую строку из композиции и превращает её в финальный отформатированный вывод.

### Протокол *граничных маркеров*
Вся архитектура держится на одной конвенции: каждый функциональный блок завершается последовательностью — "новая строка плюс пробел". Это граничный маркер блока, по которому парсер разбивает сплошную строку на отдельные блоки. Составные модули удаляют маркер из вложенных частей и добавляют свой единственный маркер в конце — так из нескольких подблоков получается один блок.

### Конвейер *фреймворка*
Рендер программы выполняет пять последовательных фаз над единой строкой, полученной из композиции:
- **Фаза № 1** — декомпозиция на блоки (строки 244–251); строка разрезается по граничным маркерам ("новая строка плюс пробел") в одномерную таблицу блочной структуры программы; каждый элемент — текст одного функционального блока;
- **Фаза № 2** — подсчёт максимума строк (строки 253–258); проход по всем блокам, определение максимального количества строк среди всех блоков; нужно для создания двумерной таблицы фиксированного размера;
- **Фаза № 3** — декомпозиция блоков на строки (строки 263–279); каждый блок разрезается по новой строке и раскладывается в двумерную таблицу — "программа"; пустые ячейки заполняются пустыми строками;
- **Фаза 4** — сборка с адаптивным рендерингом (строки 281–301); проход по двумерной таблице. Для каждой непустой строки: если она длиннее длины строки программы плюс 4 символа — вызывается алгоритм адаптивного рендеринга строк; если длиннее длины строки программы, но не более чем на 4 символа — выводится как есть, но флаг удлинения разделителя поднимается. Пустые строки сбрасывают флаг удлинения;
- **Фаза № 5** — вставка разделительных линий (строки 293–324); между блоками вставляется разделительная линия. Тип определяется диспетчером: если следующий блок начинается с `|}>-` (пользовательский ввод) — тонкая линия из дефисов; иначе — толстая из знаков равенства. Длина линии равна длине заглавия; если флаг удлинения поднят — добавляются 4 символа дефиса.

### Адаптивный рендеринг *строк*
Отдельный конвейер из четырёх фаз внутри алгоритма адаптивного рендеринга строк:
- **Фаза № 1** — декомпозиция на слова (строки 337–344); строка разбивается по пробелам в таблицу слов;
- **Фаза № 2** — подсчёт строк (строки 346–352); проход по словам с накоплением длины; когда добавление следующего слова превышает допустимую длину — счётчик строк увеличивается;
- **Фаза № 3** — сборка строк (строки 357–380); двойной цикл: внешний — по строкам, внутренний — по словам; переменная контекст индекс слова сохраняет позицию между итерациями внешнего цикла; сначала проверяется переполнение, потом — является ли слово последним;
- **Фаза № 4** — рендеринг с префиксами (строки 382–400); диспетчер выборкой определяет тип блока по первой строке и добавляет соответствующий префикс к продолжениям: `"| - "` для авторского сообщения, `"| "` для остальных блоков с `|` и `"/ "` для блока завершения; порядок веток — от специфичного к общему, что исключает ложные срабатывания.

### Парсер *текстового содержимого*
Универсальный алгоритм-счётчик с четырьмя режимами, управляемыми через таблицу-константу типов парсинга. Парсинг блоков — по граничному маркеру ("новая строка плюс пробел"), парсинг строк — по новой строке, парсинг слов — по пробелу (с обработкой последнего слова без пробела), парсинг пользовательских вводов — по маркеру "`\ввод`". Во всех случаях принцип один: найти разделитель, вырезать фрагмент, увеличить счётчик.

### Контрактная система
Алгоритмы рендеринга защищены контрактами "`дано`"/"`надо`": предусловия проверяют минимальную длину входных данных, постусловия — минимальную длину результата. Константы минимальных длин вынесены в конфигурацию в шапке файла. Утверждение в рендере заглавия дополнительно проверяет инвариант после вычислений.

## Точки *пользовательского интереса*:
1. В шапке документа представлены константы конфигурации для указания информации об авторе, наименования и описания программы, а также для указания желаемой длины строки программы;
2. Пользовательская композиция функциональных блоков основного содержимого для сборки программы (строки 77-79);
3. Место для алгоритмов пользовательской логики отведено в строках 81-82;
4. Необходимые для логики обработки пользовательских вводов переменные и таблицы объявляются после выполнения рендеринга программы до вывода результата — интерфейса программы в консоли (строки 41-42);
5. Место для реализации логики обработки пользовательских вводов отведено после обработки функциональных блоков и формирования таблицы пользовательских ответов в строках 67-68.

## Методика программирования
> [!WARNING]
> Вызов алгоритма пользовательской композиции в алгоритме-точке входа в программу может привести к ошибке выполнения задуманной логики.
 
Для вывода результатов определённых вычислений на основе пользовательского ввода рекомендуется использовать расстановку маркеров (синтаксис: `\[содержание маркера через дефис в случае использования нескольких слов]`) в композиции с последующей заменой маркера на желаемый результат в алгоритме-точке входа в программу.
